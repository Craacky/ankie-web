## HTTP и HTTPS

### Вопрос: Что такое HTTP? Что такое HTTPS?

**Ответ:** 

HTTP (HyperText Transfer Protocol) - это протокол прикладного уровня для передачи гипертекста. Это основа передачи данных в вебе, работающий поверх TCP/IP. HTTP позволяет браузеру и веб-серверу обмениваться данными посредством запросов и ответов.

HTTPS (HTTP Secure) - это защищенная версия HTTP, в которой используется шифрование (SSL/TLS) для обеспечения безопасности передаваемых данных. HTTPS шифрует все данные, что предотвращает перехват и модификацию информации третьими лицами.

**Основные отличия:**
- HTTP использует порт 80, HTTPS - порт 443
- HTTPS защищает конфиденциальность и целостность данных
- HTTPS требует SSL-сертификат от центра сертификации

## TCP и UDP

### Вопрос: Основные отличия TCP и UDP

**Ответ:**

TCP (Transmission Control Protocol) и UDP (User Datagram Protocol) - это транспортные протоколы, работающие поверх IP.

**TCP:**
- Надежный протокол с установлением соединения
- Гарантирует доставку пакетов в правильном порядке
- Использует подтверждения и повторные передачи
- Медленнее, чем UDP
- Примеры использования: HTTP, HTTPS, FTP, SMTP

**UDP:**
- Ненадежный протокол без установления соединения
- Пакеты могут приходить в неправильном порядке или теряться
- Без подтверждений и повторных передач
- Быстрее, чем TCP
- Примеры использования: VoIP, онлайн-игры, DNS

## REST API

### Вопрос: Что такое REST API?

**Ответ:**

REST (Representational State Transfer) - это архитектурный стиль для разработки распределенных систем, особенно веб-сервисов. REST API (Representational State Transfer Application Programming Interface) - это интерфейс, который следует принципам REST.

**Основные характеристики REST:**
- Архитектура клиент-сервер
- Отсутствие состояния (stateless)
- Кэширование ответов
- Единообразный интерфейс
- Поддержка слоев
- Код по требованию (опционально)

REST API использует HTTP методы (GET, POST, PUT, DELETE и др.) для выполнения CRUD операций.

## Поток документа

### Вопрос: Что такое поток документа? Как его можно изменить?

**Ответ:**

Поток документа (document flow) - это способ, которым браузер отображает элементы HTML на веб-странице в обычном порядке - сверху вниз и слева направо.

**Способы изменения потока документа:**

1. **CSS Positioning:**
   - `position: static` (по умолчанию)
   - `position: relative` - элемент смещается относительно своего обычного положения
   - `position: absolute` - элемент позиционируется относительно ближайшего позиционированного родителя
   - `position: fixed` - элемент позиционируется относительно окна браузера
   - `position: sticky` - элемент ведет себя как обычный, пока не достигает определенной позиции

2. **Float:**
   - `float: left/right` - элементы обтекают плавающий элемент

3. **Flexbox:**
   - `display: flex` - создает гибкий контейнер

4. **CSS Grid:**
   - `display: grid` - создает двумерную систему макета

5. **Display:**
   - `display: inline`, `display: block`, `display: inline-block`

## Layout, Painting и Compositing

### Вопрос: Разница между layout, painting и compositing?

**Ответ:**

Это три различных этапа в процессе отображения веб-страницы в браузере:

**Layout (компоновка):**
- Процесс вычисления геометрических свойств элементов (размеры, позиции)
- Определяет, где и как будет размещаться каждый элемент
- Происходит после изменения DOM или CSS

**Painting (отрисовка):**
- Процесс заполнения пикселей (раскраска элементов)
- Включает отрисовку текста, цвета, изображений, границ и т.д.
- Происходит после Layout

**Compositing (композиция):**
- Процесс составления отдельных слоев в правильном порядке
- Обработка таких эффектов, как z-index, opacity, transforms
- Использует аппаратное ускорение для оптимизации

## Core Web Vitals

### Вопрос: Что такое Core Web Vitals? Какие основные метрики туда входят?

**Ответ:**

Core Web Vitals - это набор метрик, разработанных Google для оценки пользовательского опыта веб-сайтов. Это три ключевых показателя:

**Largest Contentful Paint (LCP):**
- Время загрузки самого большого видимого элемента на странице
- Хороший результат: ≤ 2.5 секунд

**First Input Delay (FID):**
- Время между первым взаимодействием пользователя и ответом браузера
- Хороший результат: ≤ 100 миллисекунд

**Cumulative Layout Shift (CLS):**
- Степень неожиданного сдвига содержимого
- Хороший результат: ≤ 0.1

Эти метрики влияют на SEO-рейтинг в поисковой выдаче Google.

## Принципы REST

### Вопрос: Принципы REST(5)

**Ответ:**

Пять основных принципов REST (Representational State Transfer):

1. **Client-Server Architecture (Архитектура клиент-сервер):**
   - Четкое разделение между клиентом и сервером
   - Позволяет независимо развивать и масштабировать компоненты

2. **Stateless (Без состояния):**
   - Каждый запрос от клиента к серверу должен содержать всю необходимую информацию
   - Сервер не хранит информацию о состоянии клиента между запросами

3. **Cacheable (Кэшируемость):**
   - Ответы сервера должны быть помечены как кэшируемые или нет
   - Это позволяет клиенту повторно использовать данные и улучшает производительность

4. **Uniform Interface (Единообразный интерфейс):**
   - Простое и последовательное взаимодействие между компонентами
   - Основано на HTTP методах (GET, POST, PUT, DELETE и др.)

5. **Layered System (Многоуровневая система):**
   - Клиент может не знать, взаимодействует ли он с конечным сервером или промежуточным
   - Позволяет создавать масштабируемые архитектуры

## OSI модель

### Вопрос: Что такое OSI модель?

**Ответ:**

OSI (Open Systems Interconnection) модель - это концептуальная модель, описывающая, как сетевые протоколы взаимодействуют в сетях. Она разделена на 7 уровней:

**Уровень 7 - Прикладной (Application):**
- Взаимодействие приложений с сетью
- Примеры: HTTP, FTP, SMTP

**Уровень 6 - Представления (Presentation):**
- Преобразование данных в нужный формат
- Шифрование, сжатие данных

**Уровень 5 - Сеансовый (Session):**
- Управление сеансами связи
- Установление, поддержание и завершение сеансов

**Уровень 4 - Транспортный (Transport):**
- Надежная передача данных
- Примеры: TCP, UDP

**Уровень 3 - Сетевой (Network):**
- Маршрутизация и логическая адресация
- Примеры: IP, ICMP

**Уровень 2 - Канальный (Data Link):**
- Физическая адресация
- Примеры: Ethernet, Wi-Fi

**Уровень 1 - Физический (Physical):**
- Передача битов по физическим средам
- Примеры: кабели, радиоволны

## Протокол UDP

### Вопрос: Что такое протокол UDP?

**Ответ:**

UDP (User Datagram Protocol) - это простой транспортный протокол без установления соединения. Он отправляет данные в виде отдельных пакетов (датаграмм) без гарантии доставки, порядка или повторных передач.

**Характеристики UDP:**
- Не требует установления соединения
- Быстрее, чем TCP
- Без подтверждений доставки
- Без контроля потока и перегрузки
- Пакеты могут теряться или приходить в неправильном порядке

**Преимущества:**
- Минимальная задержка
- Меньше накладных расходов
- Подходит для реального времени

**Примеры использования:**
- DNS запросы
- VoIP (голосовая связь)
- Онлайн-игры
- Трансляции

## Протокол TCP

### Вопрос: Что такое TCP, основные характеристики?

**Ответ:**

TCP (Transmission Control Protocol) - это надежный транспортный протокол с установлением соединения. Он обеспечивает надежную и упорядоченную доставку данных между приложениями.

**Основные характеристики TCP:**
- Соединение устанавливается через трехэтапное рукопожатие (SYN, SYN-ACK, ACK)
- Гарантированная доставка пакетов
- Пакеты доставляются в правильном порядке
- Контроль перегрузки и управления потоком
- Повторная передача потерянных пакетов
- Контрольные суммы для проверки целостности данных

**Преимущества:**
- Надежность доставки данных
- Правильный порядок пакетов
- Контроль целостности данных

**Недостатки:**
- Высокие накладные расходы
- Медленнее, чем UDP
- Потенциальная задержка из-за подтверждений

## IP

### Вопрос: Что такое IP?

**Ответ:**

IP (Internet Protocol) - это сетевой протокол, отвечающий за адресацию и маршрутизацию пакетов данных в интернете. IP определяет, как пакеты должны перемещаться от источника к получателю.

**Основные характеристики IP:**
- Логическая адресация (IP-адреса)
- Маршрутизация пакетов
- Фрагментация и сборка пакетов
- Ненадежная доставка (без гарантии)

**Версии IP:**
- IPv4: 32-битные адреса (например, 192.168.1.1)
- IPv6: 128-битные адреса (например, 2001:0db8:85a3:0000:0000:8a2e:0370:7334)

IPv4 использует 4 байта для адреса, а IPv6 - 16 байт. IPv6 был разработан для решения проблемы нехватки IPv4 адресов.

## Host и Domain

### Вопрос: Разница между host и domain?

**Ответ:**

**Host (хост):**
- Уникальное имя компьютера или устройства в сети
- Может включать поддомены (например, www.example.com)
- Связано с IP-адресом через DNS
- Примеры: www.google.com, mail.example.com

**Domain (домен):**
- Область в иерархической системе доменных имен
- Часть полного доменного имени
- Может содержать поддомены и домены верхнего уровня
- Примеры: google.com, example.com

**Различие:**
- Домен - это часть иерархии (например, example.com)
- Хост - это конкретное устройство или сервис в домене (например, www.example.com)
- Хост может быть частью домена
- Домен может содержать несколько хостов

## URI и URL

### Вопрос: Разница между URI и URL?

**Ответ:**

**URI (Uniform Resource Identifier):**
- Это строка, которая идентифицирует ресурс
- Включает как URL, так и URN
- Более общее понятие
- Пример: http://example.com/page#section

**URL (Uniform Resource Locator):**
- Это подкатегория URI, которая указывает местоположение ресурса
- Описывает, как получить доступ к ресурсу
- Включает протокол, домен, путь и т.д.
- Пример: https://www.example.com/index.html

**Различие:**
- Все URL являются URI
- Не все URI являются URL
- URL указывает местоположение и метод доступа
- URI может просто идентифицировать ресурс

## Хранение токенов

### Вопрос: В чём безопаснее хранить токен - в localStorage или cookies? Почему?

**Ответ:**

**Cookies безопаснее localStorage для хранения токенов** по следующим причинам:

**Проблемы с localStorage:**
- Уязвимость к XSS (Cross-Site Scripting) атакам
- JavaScript может получить доступ к localStorage
- Данные остаются в браузере до их удаления
- Нет встроенной защиты от CSRF

**Преимущества Cookies с правильной настройкой:**
- HttpOnly флаг предотвращает доступ к куке через JavaScript
- SameSite флаг защищает от CSRF атак
- Secure флаг гарантирует передачу только по HTTPS
- Куки автоматически удаляются при закрытии сеанса (для сессионных кук)

**Рекомендации:**
- Использовать HttpOnly, Secure и SameSite куки для токенов доступа
- Для refresh токенов использовать только куки с HttpOnly
- Access токены могут храниться в памяти приложения для дополнительной безопасности

## Сценарий работы HTTP протокола

### Вопрос: Примерный сценарий работы HTTP-протокола?

**Ответ:**

**Сценарий работы HTTP протокола:**

1. **DNS разрешение:**
   - Браузер преобразует доменное имя в IP-адрес через DNS серверы

2. **Установление TCP соединения:**
   - Выполняется трехэтапное рукопожатие (SYN, SYN-ACK, ACK)
   - Открывается TCP-соединение с веб-сервером

3. **Отправка HTTP запроса:**
   - Клиент отправляет HTTP запрос (GET, POST и т.д.)
   - Запрос включает метод, путь, заголовки, и тело (для POST)

4. **Обработка запроса сервером:**
   - Сервер обрабатывает запрос
   - Выполняет необходимые действия (доступ к БД, логика и т.д.)

5. **Формирование HTTP ответа:**
   - Сервер формирует HTTP ответ с кодом состояния
   - Ответ включает заголовки и тело ответа

6. **Отправка ответа:**
   - Сервер отправляет ответ клиенту

7. **Закрытие соединения:**
   - В HTTP/1.0 соединение закрывается после каждого запроса
   - В HTTP/1.1 и HTTP/2 соединение может быть повторно использовано

## Идентификация, аутентификация, авторизация

### Вопрос: Разница между идентификацией, аутентификацией, авторизацией?

**Ответ:**

**Идентификация (Identification):**
- Процесс предоставления уникального идентификатора
- Заявление "кто я"
- Пример: ввод имени пользователя или email

**Аутентификация (Authentication):**
- Процесс проверки подлинности пользователя
- Доказательство, что пользователь - это действительно он
- Пример: ввод пароля, проверка отпечатка пальца

**Авторизация (Authorization):**
- Процесс определения, какие права доступа имеет удостоверенный пользователь
- Определение того, что пользователь может делать
- Пример: разрешение доступа к определенным функциям или данным

**Простой пример:**
- Идентификация: "Я Джон"
- Аутентификация: "Вот мой ID, доказывающий, что я Джон"
- Авторизация: "Как Джон, я могу получить доступ к своей почте, но не к чужой"

## POST vs GET запросы

### Вопрос: В каком случае понадобится использовать POST запрос вместо GET, даже если мы не хотим ничего менять на сервере?

**Ответ:**

POST запрос может использоваться вместо GET в следующих случаях:

1. **Большой объем данных:**
   - GET ограничен длиной URL (обычно 2048 символов)
   - POST может передавать большие объемы данных в теле запроса

2. **Конфиденциальные данные:**
   - GET передает данные в URL, что может быть видно в истории браузера, логах сервера и т.д.
   - POST передает данные в теле запроса, сохраняя конфиденциальность

3. **Поиск или фильтрация:**
   - Поисковые формы часто используют POST для отправки сложных критериев поиска
   - Это позволяет отправлять сложные структурированные данные

4. **Кэширование:**
   - GET запросы кэшируются браузерами и промежуточными серверами
   - POST запросы не кэшируются, что может быть желательно для актуальных данных

5. **Идемпотентность:**
   - Хотя GET идемпотентен, в некоторых сценариях может быть предпочтительно использовать POST для операций, которые не должны быть кэшированы или повторно выполнены при случайном перезапуске

## Структура HTTP запроса

### Вопрос: Из чего состоит HTTP запрос?

**Ответ:**

HTTP запрос состоит из следующих частей:

**1. Стартовая строка (Request Line):**
- HTTP метод (GET, POST, PUT, DELETE и т.д.)
- Путь к ресурсу (URI)
- Версия HTTP протокола
- Пример: `GET /index.html HTTP/1.1`

**2. Заголовки (Headers):**
- Именованные значения, передающие дополнительную информацию
- Каждый заголовок на отдельной строке
- Примеры: Host, User-Agent, Content-Type, Authorization

**3. Пустая строка:**
- Разделитель между заголовками и телом

**4. Тело запроса (Optional Body):**
- Данные, отправляемые с запросом
- Используется в POST, PUT, PATCH запросах
- Примеры: данные формы, JSON, XML

**Пример полного HTTP запроса:**
```
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 52
Authorization: Bearer token12345

{
  "name": "John Doe",
  "email": "john@example.com"
}
```

## Access/Refresh токены

### Вопрос: Зачем нужны access/refresh токены: создание, передача, использование, обновление

**Ответ:**

**Зачем нужны access/refresh токены:**
- Для безопасной аутентификации без хранения паролей на клиенте
- Для снижения рисков безопасности при компрометации токенов
- Для управления сессиями без хранения состояния на сервере

**Создание:**
- Access токен создается при успешной аутентификации
- Refresh токен также создается и сохраняется у клиента
- Access токен имеет короткий срок действия (например, 15-60 минут)
- Refresh токен имеет более длительный срок действия (например, 7-30 дней)

**Передача:**
- Access токен передается в заголовке Authorization: Bearer токен
- Refresh токен может храниться в HttpOnly cookie или безопасно в памяти

**Использование:**
- Access токен используется для доступа к защищенным ресурсам
- Для каждого API запроса включается access токен

**Обновление:**
- Когда access токен истекает, клиент использует refresh токен для получения нового access токена
- На сервере отправляется запрос с refresh токеном
- Сервер проверяет refresh токен и выдает новый access токен
- В некоторых системах также обновляется refresh токен

## Термины в безопасности

### Вопрос: Что такое идентификация, аутентификация, авторизация, регистрация, верификация?

**Ответ:**

**Идентификация:**
- Процесс предоставления уникального идентификатора
- Заявление "кто я"
- Пример: ввод имени пользователя

**Аутентификация:**
- Процесс проверки подлинности пользователя
- Доказательство, что пользователь - это действительно он
- Пример: ввод пароля, проверка отпечатка пальца

**Авторизация:**
- Процесс определения, какие права доступа имеет удостоверенный пользователь
- Определение того, что пользователь может делать
- Пример: разрешение доступа к определенным функциям

**Регистрация:**
- Процесс создания учетной записи пользователя
- Создание идентификатора и установка аутентификационных данных
- Пример: заполнение формы регистрации с паролем

**Верификация:**
- Процесс подтверждения достоверности предоставленной информации
- Проверка, например, email или номера телефона
- Пример: подтверждение по ссылке из email

## Работа с Access/Refresh токенами

### Вопрос: Как работать с access/refresh токенами?

**Ответ:**

**Работа с access/refresh токенами включает следующие этапы:**

**1. Авторизация:**
- Пользователь отправляет учетные данные (имя/пароль)
- Сервер проверяет данные и создает пару токенов

**2. Хранение:**
- Access токен может храниться в памяти приложения
- Refresh токен должен храниться безопасно, обычно в HttpOnly cookie

**3. Использование access токена:**
- Добавлять в заголовок Authorization: Bearer токен
- Использовать для доступа к защищенным API
- Проверять срок действия перед каждым запросом

**4. Обновление токенов:**
- Когда access токен истекает, использовать refresh токен для получения новых
- Отправить refresh токен на специальный endpoint
- Получить новую пару токенов

**5. Обработка ошибок:**
- При ошибке refresh токена (например, истек) перенаправить на страницу входа
- Обработать случай, когда refresh токен скомпрометирован

**6. Безопасность:**
- Использовать HttpOnly, Secure и SameSite флаги для refresh токенов
- Обновлять refresh токен при каждом использовании (rotation)
- Ограничивать количество использований refresh токена

## HTTP протокол

### Вопрос: Расскажи про HTTP: что такое, основные характеристики, структура ответа и запроса, статусы ответов какие-нибудь

**Ответ:**

**HTTP (HyperText Transfer Protocol)** - это протокол прикладного уровня для передачи гипертекста. Это основа передачи данных в вебе, работающий поверх TCP/IP.

**Основные характеристики HTTP:**
- Протокол клиент-сервер
- Не поддерживает состояние (stateless)
- Работает поверх TCP (обычно на порту 80)
- Использует методы запросов (GET, POST, PUT, DELETE и др.)

**Структура HTTP запроса:**
```
МЕТОД URI HTTP/Версия\r\n
Заголовки: Значение\r\n
\r\n
Тело запроса (опционально)
```

**Структура HTTP ответа:**
```
HTTP/Версия Код_статуса Сообщение_статуса\r\n
Заголовки: Значение\r\n
\r\n
Тело ответа (опционально)
```

**Категории HTTP статусов:**
- **1xx (Informational)**: Промежуточные статусы (100 Continue, 101 Switching Protocols)
- **2xx (Success)**: Успешные ответы (200 OK, 201 Created, 204 No Content)
- **3xx (Redirection)**: Перенаправления (301 Moved Permanently, 302 Found, 304 Not Modified)
- **4xx (Client Error)**: Ошибки клиента (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 405 Method Not Allowed)
- **5xx (Server Error)**: Ошибки сервера (500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout)

**Часто используемые статусы:**
- 200 OK: Успешный запрос
- 201 Created: Ресурс успешно создан
- 404 Not Found: Ресурс не найден
- 500 Internal Server Error: Внутренняя ошибка сервера
