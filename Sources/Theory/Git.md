## 1. Что такое Git?

**Ответ:** Git - это распределенная система контроля версий, разработанная Линусом Торвальдсом. Она позволяет отслеживать изменения в файлах и координировать работу нескольких разработчиков над проектами. Git хранит историю изменений в виде графа коммитов, что позволяет легко переключаться между разными версиями кода, создавать ветки и объединять изменения.

## 2. Что такое git flow?

**Ответ:** Git Flow - это стратегия ветвления в Git, предложенная Винсентом Дриессеном. Она определяет структуру ветвления и процесс объединения изменений:

**Основные ветки:**
- `master/main` - основная ветка для продакшн-версий, содержит только стабильный код, каждое изменение соответствует релизу
- `develop` - интеграционная ветка для текущей разработки, в нее сливаются все feature-ветки

**Вспомогательные ветки:**
- `feature` - ветки для разработки новых функций, ответвляются от develop и сливаются обратно в develop
- `release` - ветки для подготовки релизов, ответвляются от develop и сливаются в master и develop
- `hotfix` - ветки для срочных исправлений продакшн-багов, ответвляются от master и сливаются в master и develop

**Преимущества Git Flow:**
- Четкая структура разработки
- Понятные этапы жизненного цикла релизов
- Легкость отслеживания изменений
- Возможность параллельной разработки
- Простота отката изменений
- Поддержка параллельных релизов

**Недостатки:**
- Сложность для новичков
- Дополнительная сложность при частых релизах
- Больше веток для управления
- Может быть избыточным для небольших проектов

**Команды для работы с Git Flow (с использованием расширения git-flow):**
```bash
git flow feature start <имя>
git flow feature finish <имя>
git flow release start <версия>
git flow release finish <версия>
git flow hotfix start <версия>
git flow hotfix finish <версия>
```

## 3. В чем отличие Git от SVN?

**Ответ:**
- **Git** - распределенная система контроля версий, **SVN** - централизованная
- В Git вся история доступна локально, в SVN требуется соединение с сервером
- Git работает быстрее при локальных операциях
- В Git коммиты представлены как снимки, в SVN как дельты
- Git лучше поддерживает работу с ветками

## 4. Какие основные команды Git вам известны?

**Ответ:**
- `git init` - инициализация репозитория
- `git clone` - клонирование репозитория
- `git add` - добавление файлов в индекс
- `git commit` - создание коммита
- `git push` - отправка изменений на удаленный репозиторий
- `git pull` - получение и объединение изменений
- `git fetch` - получение изменений с удаленного репозитория
- `git merge` - объединение веток
- `git rebase` - перенос изменений
- `git status` - статус файлов
- `git log` - история коммитов

## 5. Как создать коммит с сообщением в Git?

**Ответ:** Для создания коммита с сообщением используется команда:
```bash
git commit -m "Краткое описание изменений"
```

## 6. Что делает команда `git clone`?

**Ответ:** Команда `git clone` создает локальную копию удаленного репозитория. Она копирует все файлы, историю коммитов, ветки и теги из удаленного репозитория на локальный компьютер.

## 7. Что делает команда `git config`?

**Ответ:** Команда `git config` используется для настройки параметров Git. Можно задавать имя пользователя, email, редактор по умолчанию и другие настройки:
```bash
git config --global user.name "Имя пользователя"
git config --global user.email "email@example.com"
```

## 8. Как убрать файлы из индекса?

**Ответ:** Для удаления файлов из индекса используются следующие команды:
```bash
git reset HEAD <имя_файла>  # убирает конкретный файл из индекса
git reset                   # убирает все файлы из индекса
```

## 9. Преимущества Git?

**Ответ:**
- Распределенная архитектура
- Быстродействие
- Поддержка ветвления и слияния
- Возможность работы оффлайн
- История изменений
- Гибкие стратегии слияния

## 10. Что содержит объект класса Git?

**Ответ:** Git хранит данные в виде 4 типов объектов:
- **Blob** - содержит содержимое файла
- **Tree** - содержит список файлов и подкаталогов
- **Commit** - содержит указатель на дерево, родительские коммиты и метаданные
- **Tag** - содержит ссылку на коммит

## 11. Что такое репозиторий Git?

**Ответ:** Репозиторий Git - это папка, в которой Git хранит все данные проекта, включая историю изменений, ветки, теги и другие метаданные. Внутри папки `.git` находится вся информация системы контроля версий.

## 12. Индекс в Git?

**Ответ:** Индекс (или staging area) - это промежуточное состояние между рабочей директорией и репозиторием. Это список файлов, которые будут включены в следующий коммит. Команда `git add` добавляет файлы в индекс.

## 13. Содержание репозитория?

**Ответ:** Репозиторий Git содержит:
- Рабочую директорию (working directory)
- Область хранения (staging area)
- Локальный репозиторий (local repository)
- Все коммиты, ветки, теги и историю изменений

## 14. Что делает `git stash`?

**Ответ:** Команда `git stash` временно сохраняет несохраненные изменения в рабочей директории. Это полезно, когда нужно переключить ветку, не коммитя текущие изменения. Позже можно восстановить сохраненные изменения командой `git stash pop`.

## 15. Что такое cherry-pick в Git?

**Ответ:** `git cherry-pick` - это команда, которая позволяет применить один или несколько коммитов из другой ветки к текущей ветке. Это полезно, когда нужно применить конкретные изменения к другой ветке без полного слияния.

**Синтаксис:**
```bash
git cherry-pick <commit-hash>                    # применить один коммит
git cherry-pick <commit1> <commit2> <commit3>    # применить несколько коммитов
git cherry-pick <commit1>..<commit3>             # диапазон коммитов (не включая первый)
git cherry-pick <commit1>^..<commit3>            # диапазон с включением первого коммита
git cherry-pick -n <commit-hash>                 # применить изменения без коммита
```

**Особенности:**
- Создает новый коммит с тем же содержимым, но новым хешем
- Может вызвать конфликты слияния, которые нужно разрешить вручную
- Используется для "переноса" изменений между ветками
- Полезен для исправления багов в разных ветках
- Может использоваться с опцией `-x` для указания оригинального коммита
- Рекомендуется использовать с опцией `--no-commit` для избежания автоматического коммита

**Примеры использования:**
- Применение срочного исправления из develop ветки в hotfix-ветку
- Перенос независимой функциональности из одной ветки в другую
- Восстановление изменений из откаченной ветки
- Исправление ошибок в разных версиях продукта

**Разрешение конфликтов:**
Если возникают конфликты, их нужно разрешить вручную и затем выполнить:
```bash
git add <файлы>
git cherry-pick --continue
```

Или отменить cherry-pick:
```bash
git cherry-pick --abort
```

**Практические рекомендации:**
- Использовать осторожно с публичными коммитами
- Рекомендуется использовать с флагом `-x` для отслеживания происхождения изменений
- Удобно использовать в связке с интерактивным rebase для сложного переноса изменений

## 16. Что такое `git merge`?

**Ответ:** `git merge` объединяет изменения из другой ветки в текущую ветку. Создает коммит слияния, который содержит изменения из обеих веток. Это безопасный способ объединения веток.

## 17. Что такое `git rebase`?

**Ответ:** `git rebase` переносит изменения из одной ветки на вершину другой ветки. В отличие от merge, он не создает коммит слияния, а переписывает историю, делая ее линейной.

## 18. В чем разница между merge и rebase?

**Ответ:**

| Характеристика | Merge | Rebase |
|----------------|-------|--------|
| История | Создает коммит слияния, сохраняя логику разработки | Переписывает историю, делая ее линейной |
| Безопасность | Безопасен при работе с публичными ветками | Может быть опасен для общих веток |
| Влияние на ветки | Изменяет только целевую ветку | Может изменять коммиты в обеих ветках |
| Конфликты | Обрабатываются один раз при слиянии | Обрабатываются для каждого коммита при переносе |
| Сложность | Простой и понятный процесс | Более сложный процесс |
| Хеши коммитов | Сохраняются оригинальные хеши | Хеши изменяются при переносе |

**Merge (слияние):**
- Создает специальный коммит слияния (merge commit), содержащий изменения из обеих веток
- Сохраняет полную историю разработки с ветвлениями
- Не изменяет существующую историю коммитов
- Предпочтителен для объединения с публичными ветками
- Создает более сложную, но честную историю
- Команда: `git merge <ветка>`

**Rebase (переоснование):**
- Берет изменения из ветки и "переосновывает" их поверх целевой ветки
- Создает линейную, более чистую историю без коммитов слияния
- Переписывает хеши коммитов
- Полезен для подготовки clean-ветки перед pull request
- Может переносить изменения коммит за коммитом
- Команда: `git rebase <ветка>`

**Когда использовать:**
- **Merge** - для объединения публичных веток, при работе в команде, когда важна сохранность истории разработки
- **Rebase** - для подготовки локальных изменений перед отправкой, для синхронизации с master, при работе с локальными ветками

**Дополнительные особенности:**
- Rebase может использоваться в интерактивном режиме (`git rebase -i`) для редактирования, объединения или удаления коммитов
- После rebase, если ветка была отправлена на удаленный репозиторий, потребуется принудительная отправка (`git push --force`)
- Merge может вызывать конфликты только один раз, rebase может требовать разрешения конфликтов для каждого коммита
- Rebase рекомендуется использовать только с локальными ветками, которые еще не были опубликованы

## 19. Какова функция `git push` в GIT?

**Ответ:** Команда `git push` отправляет локальные изменения в удаленный репозиторий. Она обновляет удаленные ветки соответствующими коммитами из локальных веток.

## 20. Каковы отличия между `git pull` и `git fetch`?

**Ответ:**
- `git fetch` просто получает изменения с удаленного репозитория, не объединяя их
- `git pull` получает и автоматически объединяет изменения с локальной веткой
- fetch безопаснее, так как не меняет локальное состояние

## 21. Что такое pull request (также известен как merge request)?

**Ответ:** Pull Request (PR) или Merge Request (MR) - это запрос на включение изменений из одной ветки в другую. Он позволяет просматривать код, обсуждать изменения и проводить рецензирование перед объединением. Используется в системах Git-хостинга (GitHub, GitLab и др.).

## 22. Как реализовать CI/CD для Git репозитория?

**Ответ:** CI/CD (Continuous Integration/Continuous Deployment) - это методология разработки программного обеспечения, при которой изменения кода автоматически тестируются, собираются и развертываются.

**Continuous Integration (CI) - Непрерывная интеграция:**
- Автоматическое тестирование при каждом пуше в репозиторий
- Проверка стиля кода, линтинг
- Запуск модульных и интеграционных тестов
- Сборка приложения
- Проверка безопасности кода и зависимостей
- Создание артефактов сборки

**Continuous Delivery/Deployment (CD) - Непрерывная доставка/развертывание:**
- Автоматическое развертывание в тестовую среду
- Автоматическое развертывание в продакшн при прохождении всех проверок
- Создание образов (Docker)
- Обновление инфраструктуры
- Постепенное развертывание (blue-green, canary)

**Реализация CI/CD:**

1. **Выбор платформы CI/CD:**
   - GitHub Actions (встроенный в GitHub)
   - GitLab CI/CD (встроенный в GitLab)
   - Jenkins (самоhosted, гибкая настройка)
   - CircleCI (облачное решение)
   - Travis CI (облачное решение)
   - Bitbucket Pipelines
   - Azure DevOps
   - AWS CodePipeline

2. **Конфигурационные файлы:**
   - GitHub Actions: `.github/workflows/ci.yml`
   - GitLab CI: `.gitlab-ci.yml`
   - Jenkins: `Jenkinsfile` (Declarative Pipeline)
   - CircleCI: `.circleci/config.yml`

3. **Типичный CI/CD пайплайн:**
   ```yaml
   # Пример для GitHub Actions
   name: CI/CD Pipeline

   on:
     push:
       branches: [ main, develop ]
     pull_request:
       branches: [ main ]

   jobs:
     build:
       runs-on: ubuntu-latest
       steps:
         - name: Checkout code
           uses: actions/checkout@v3

         - name: Setup Node.js
           uses: actions/setup-node@v3
           with:
             node-version: '18'

         - name: Install dependencies
           run: npm install

         - name: Run tests
           run: npm test

         - name: Run linting
           run: npm run lint

         - name: Build
           run: npm run build

         - name: Upload build artifacts
           uses: actions/upload-artifact@v3
           with:
             name: build-files
             path: dist/

     deploy-staging:
       needs: build
       runs-on: ubuntu-latest
       if: github.ref == 'refs/heads/develop'
       steps:
         - name: Deploy to staging
           run: echo "Deploying to staging environment"

     deploy-production:
       needs: build
       runs-on: ubuntu-latest
       if: github.ref == 'refs/heads/main'
       steps:
         - name: Deploy to production
           run: echo "Deploying to production environment"
   ```

4. **Этапы CI/CD пайплайна:**
   - **Checkout:** получение кода из репозитория
   - **Dependencies:** установка зависимостей
   - **Build:** сборка приложения
   - **Test:** запуск тестов (модульных, интеграционных, e2e)
   - **Lint:** проверка стиля кода
   - **Security:** сканирование уязвимостей
   - **Deploy:** развертывание в нужную среду

5. **Триггеры:**
   - При пуше в определенные ветки
   - При создании Pull Request
   - По расписанию (scheduled jobs)
   - Вручную (manual triggers)
   - При событиях в других сервисах

6. **Артефакты:**
   - Собранные образы (Docker images)
   - Результаты тестов
   - Отчеты о покрытии кода
   - Отчеты о безопасности
   - Собранные файлы приложения

7. **Окружения:**
   - Development: для разработчиков
   - Staging: для предварительного тестирования
   - Production: продакшн среда

8. **Безопасность в CI/CD:**
   - Хранение секретов (токены, пароли) в зашифрованном виде
   - Проверка безопасности кода (SAST)
   - Сканирование зависимостей (SCA)
   - Проверка безопасности образов (DAST)
   - Использование безопасных практик развертывания

9. **Мониторинг и алертинг:**
   - Логирование процессов CI/CD
   - Метрики производительности
   - Алерты о сбоях
   - Визуализация прогресса

10. **Лучшие практики:**
    - Автоматизация как можно большего количества процессов
    - Быстрая обратная связь (fast feedback loops)
    - Идемпотентные скрипты развертывания
    - Тестирование в среде, максимально приближенной к продакшн
    - Использование контейнеров для консистентности окружений
    - Регулярное обновление зависимостей и инструментов
